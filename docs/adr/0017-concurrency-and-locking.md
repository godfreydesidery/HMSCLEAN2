# ADR-0017: Concurrency & locking for stock and documents
- **Status:** Proposed (Architecture phase)
- **Date:** 2026-06-02
- **Deciders:** solution-architect (reviewed by security-architect, data-architect)
- **Engagement:** Zana HMIS modernization, fresh build, no data migration

## Context
The legacy system has two confirmed concurrency defects that the fresh build must correct while preserving the business PROCESS exactly.

**Stock decrement is a check-then-act race.** Every stock-mutating path — GRN approve, dispense, store→pharmacy and pharmacy→pharmacy transfer orders — reads the current stock into a local (`double originalStock = storeItem.getStock()`), guards `if (stock < qty) throw InvalidOperationException("...less than transfer qty...")`, then `setStock(stock - qty)` and `save()` (confirmed in `StoreToPharmacyTOServiceImpl` lines 250-259, `PharmacyToPharmacyTOServiceImpl` line 248-251, `GoodsReceivedNoteServiceImpl` lines 128-148). There is no row lock and no `@Version`; under concurrent dispensing two requests read the same balance, both pass the guard, and the system oversells / drives stock negative. The legacy DOES enforce a negative-stock control (the throw guard, plus a `NegativeStockReport`) — that PROCESS must be preserved, but made race-safe. GRN approve additionally writes a `StoreStockCard` ledger row and creates batches in the same transaction; the stock balance and the card balance must stay consistent.

**Document numbers are generated by `SELECT MAX(id)+1`.** Every `requestXxxNo()` uses JPQL `getLastId()` (e.g. `SELECT MAX(grn.id) FROM GoodsReceivedNote`), increments, and formats (`GRN{yyyyMMdd}-{id}`, `LPO…`, `PRL…`, etc.; finding C). This is non-atomic: concurrent requests read the same MAX and emit duplicate numbers. Format strings (ADR-0006) must be reproduced exactly; only the generation mechanism changes.

General entity updates (status transitions, edits) also need lost-update protection, but are low-contention.

## Decision
**1. Optimistic locking by default.** The base entity carries `@Version` (a `bigint version` column). All general updates use optimistic locking; a stale write raises `OptimisticLockException`, surfaced as RFC 7807 `409 Conflict` (`type` = `urn:hmis:error:stale-entity`, per ADR-0009). This is the cheap, scalable default for everything.

**2. Pessimistic row lock for stock decrement.** Stock balance rows (`stock_balance`) and FEFO batches (`stock_batch`) are read `SELECT … FOR UPDATE` (`@Lock(PESSIMISTIC_WRITE)`) at the start of every decrement transaction (GRN approve, dispense, transfer issue). The repository exposes `lockBalance(storeUid/pharmacyUid, itemUid)` and `lockFefoForDispense(...)` (batches in expiry-ascending order, null expiry last). The negative-stock guard runs AFTER the lock is held, so the read-decrement-write is serialized per (location, item). This adopts the prior-attempt "FEFO batch dispensing with pessimistic lock" pattern verbatim and avoids retry loops on the hottest path. The lock scope is one stock row, so it does not serialize the whole transaction.

**3. Atomic, gap-tolerant document numbers via PostgreSQL `SEQUENCE`.** One named sequence per document type (`seq_grn`, `seq_lpo`, `seq_prl`, …), allocated with `nextval()` inside the transaction. The numeric portion is sequence-driven; the prefix + `yyyyMMdd` formatting stays identical. Sequences are gap-tolerant by design (a rolled-back transaction burns a number) — acceptable, because the legacy `MAX(id)+1` was already non-contiguous after deletes and the format encodes a date, not a guaranteed-dense counter. The "SPT" prefix collision (finding D) is resolved here by assigning distinct prefixes per type.

## Considered alternatives
- **Optimistic locking + retry on the stock row.** Rejected as the primary stock mechanism: dispense is the highest-contention path; a retry storm under a busy pharmacy degrades throughput and the retry semantics are harder to reason about than a short row lock. Optimistic remains the default for non-stock entities.
- **Atomic conditional UPDATE** (`UPDATE stock_balance SET qty = qty - :q WHERE item = :i AND qty >= :q`, rows-affected = 0 ⇒ insufficient). A valid race-safe option and the cleanest for a balance-only model. Rejected as primary because the legacy decrement is multi-row (balance + per-batch FEFO walk + stock-card ledger) and must stay internally consistent; a single conditional UPDATE cannot express FEFO batch selection. We may use it as an optimization for simple single-row consumable issues.
- **Application-level / advisory locks (`pg_advisory_xact_lock`).** Rejected: row-scoped `FOR UPDATE` is more precise and DB-portable for tests.
- **A `document_counter` table with `SELECT FOR UPDATE`.** Rejected in favour of native `SEQUENCE` — sequences are non-transactional, contention-free, and need no extra table; gap-tolerance is acceptable.

## Consequences
- Lost updates, oversell, and unintended negative stock are eliminated; the legacy negative-stock REFUSAL behaviour is preserved (same exception → `409`/`422`).
- Document numbers are duplicate-free under concurrency; numbers may have gaps (documented, accepted).
- A short pessimistic lock is held per (location, item) during decrement; lock ordering must be deterministic (always lock balance before batches, items in uid order within a multi-line GRN/transfer) to prevent deadlocks. A statement/lock timeout maps any deadlock to `409 Conflict` with retry guidance.
- `@Version` adds one column per table and a tiny write cost; negligible.
- Testcontainers integration tests must include concurrency cases (two parallel dispenses of last unit; two parallel `requestNo` calls) as part of the golden-master suite.

## Exact-process impact
None on business outcomes. Same negative-stock refusal, same document-number FORMATS and sequence semantics, same stock-card ledger. Only the generation/serialization mechanism changes — a correctness improvement, not a behaviour change. Numeric parity is by business result (BigDecimal per ADR-0007), not double bit-rounding.

## Implementation notes
- `@Version private long version;` on the shared base entity (hidden, never serialized — ADR-0003).
- Stock repos: `@Lock(LockModeType.PESSIMISTIC_WRITE) @Query("select b from StockBalance b where b.locationUid=:loc and b.itemUid=:item")`; set a `javax.persistence.lock.timeout` hint and a Postgres `statement_timeout` so blocked dispensers fail fast rather than hang.
- Decrement service order: acquire lock → re-read balance → assert `qty >= requested` (preserve legacy message) → decrement balance → FEFO-decrement batches → append `stock_card` row with running balance → commit.
- Document numbers: `nextval('seq_grn')` via a small `DocumentNumberService.next(DocType)` that owns prefix+date formatting; Flyway creates one sequence per type in the same migration that defines ADR-0006 formats.
- Surface `OptimisticLockException` and lock-timeout/deadlock through the global `ProblemDetail` handler with a stable `ErrorCode` (`STALE_ENTITY`, `STOCK_CONTENDED`) so the Angular client can react programmatically without string-matching (ADR-0009 lesson).
- Spring Modulith: locking is intra-module (inventory/pharmacy own their stock rows); no new cross-module edges. Decrement is triggered by billing→encounter / dispense events in the already-allowed direction.
