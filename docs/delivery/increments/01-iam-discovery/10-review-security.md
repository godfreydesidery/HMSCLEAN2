# 10-review-security

Verdict: APPROVE_WITH_NITS

Security review of Increment 01 (Identity & Access) against ADR-0006. The security delta is a net strengthening over the prior baseline and faithful to the ratified decisions. CORS is locked to an explicit, property-driven allow-list (no wildcard, allowCredentials disabled, correct for bearer tokens) and CorsIT verifies both echo and rejection. Refresh-token reuse detection returns a generic RFC-7807 401 with NO token hash / replaced_by_uid / user_uid; the revoke endpoint is non-enumerating (204 for unknown/owned/already-revoked) and ownership-checked (403 cross-user via AccessDeniedException). Every state-mutating IAM endpoint is gated with only the 26 live codes (PrivilegeGateArchTest enforces this), reads require authentication via .anyRequest().authenticated(), and the hardened privilege-assignment gate (ROLE-ALL/ADMIN-ACCESS) is present where legacy had it commented out. No literal JWT/HMAC secret exists anywhere — the SecretKeySpec key is derived from jwtProperties.secret() (a method call, so the Semgrep SecretKeySpec rule does not false-positive), and the HMAC256 grep + NoHardcodedSecretTest + Semgrep rules (MACSigner/HMAC256/SecretKeySpec/secret-named literals) all remain. BCrypt strength is 12, refresh tokens are SHA-256 hashed at rest, and the JWT carries only iss/sub/iat/exp/uid/privileges (no PHI, no pharmacy/session state). I found no weakening vs ADR-0006. The one ratified item not delivered is the V5 cost-12 admin re-hash (D-8), which was left as a commented-out TODO so the seed remains cost-10; residual risk is LOW (dev-only, publicly documented "password") but it is an unfulfilled ratified decision. Remaining items are NITs: a missing cross-user-revoke 403 test, the reuse event's performedBy=\"anonymous\", and login() recording AuditAction.CREATE. None block the PR.

## Findings

### [MEDIUM] Secrets / BCrypt seed (ADR-0006 D-8)
- file: d:\My_Works\HMS\HMSCLEAN2\backend\src\main\resources\db\migration\V5__backfill_admin_identity.sql 8-17
- issue: ADR-0006 / D-8 ratified re-hashing the cost-10 admin seed to cost-12, but V5 ships the entire re-hash UPDATE commented out (a TODO with a $2a$12$REPLACE_WITH_REAL_COST12_HASH placeholder). The seeded admin therefore remains BCrypt cost-10 (V2: $2a$10$N9qo8uLOick...), diverging from the BCryptPasswordEncoder(12) policy that BCryptPasswordEncoderTest enforces for all application-created users. Residual risk is LOW because the password is the publicly-documented dev default "password" and nothing new leaks; the operational risk is that this seed/migration is reused in a non-dev environment and the admin principal silently keeps a weaker work factor than every other account. This is a strengthening-only item, so it does not weaken ADR-0006, but it is an unfulfilled ratified decision rather than a deferred-by-design one. Recommend not blocking the PR on it but tracking it as an explicit follow-up (it is mislabeled "optional" in the V5 comment — D-8 ratified it, it is not optional).
- fix: Either (a) complete D-8: generate new BCryptPasswordEncoder(12).encode("password") and replace the placeholder so the guarded, idempotent UPDATE runs; or (b) if the team consciously defers it, change the V5 comment from "optional" to an explicit reference to D-8 with a tracked follow-up ticket id, and confirm the admin bootstrap re-hashes on first login in a later increment. Do not let the cost-10 seed reach any shared/staging environment.

### [NIT] Test coverage — revoke ownership (CR-10)
- file: d:\My_Works\HMS\HMSCLEAN2\backend\src\test\java\com\otapp\hmis\iam\TokenRevokeIT.java n/a
- issue: The ownership check in AuthenticationService.revoke (throws AccessDeniedException → 403 when the token belongs to a different user) is correct, but the build-spec §7 test revoke_crossUserToken_returns403 is not present. TokenRevokeIT covers 204/idempotent/non-enumerating/unauthenticated, but the cross-user 403 branch — the actual security control protecting one user's session from another authenticated user — is untested. This is the most security-relevant branch of the revoke flow and should not ship unverified.
- fix: Add a test that logs in as admin to obtain admin's refresh token, then calls POST /api/v1/auth/token/revoke with a second authenticated user's Bearer token (e.g. a TestJwtFactory token for a different username) presenting admin's refresh token, and asserts 403 urn:hmis:error:forbidden and that admin's token remains live (subsequent refresh still works).

### [NIT] Audit trail consistency
- file: d:\My_Works\HMS\HMSCLEAN2\backend\src\main\java\com\otapp\hmis\iam\application\AuthenticationService.java 83, 108, 171
- issue: login() records auditRecorder.record("iam.User", uid, AuditAction.CREATE, username) for a successful authentication, and both the reuse branch and revoke record AuditAction.DELETE on "iam.RefreshToken". The AuditAction taxonomy (CREATE/READ/UPDATE/DELETE) has no LOGIN/REVOKE action, so authentication and token-lifecycle events are shoehorned into CREATE/DELETE. This produces a semantically misleading append-only trail (a login appears as a CREATE of the user record). It is not a weakening of any clinical/financial/identity mutation audit and does not block the PR, but it dilutes the forensic value of the identity audit trail. Flagging for the audit-trail-requirements alignment with data-architect (audit action taxonomy may need AUTHENTICATE/REVOKE actions in a later increment).
- fix: Track for the audit-trail-requirements doc: either extend AuditAction with authentication/token lifecycle values (e.g. AUTHENTICATE, REVOKE) or document explicitly that login maps to CREATE and token revocation maps to DELETE on iam.RefreshToken. No code change required this increment; record the convention so downstream audit consumers parse it correctly.

### [NIT] Reuse-detection forensics
- file: d:\My_Works\HMS\HMSCLEAN2\backend\src\main\java\com\otapp\hmis\iam\application\AuthenticationService.java 110
- issue: On reuse detection the RefreshTokenReuseDetectedEvent is published with performedBy="anonymous". The /token/refresh endpoint is in permitAll, so there is genuinely no authenticated principal — "anonymous" is defensible and the userUid (the token's owner) is captured, which is the security-relevant identifier. However, the literal string differs from the SYSTEM convention used elsewhere (UserAdminService.currentActor() returns "SYSTEM" for the unauthenticated case), so the audit/log trail uses two different sentinels for "no authenticated actor". Minor consistency nit; the WARN-level reuse log itself is correct and leaks no token value.
- fix: Optionally align the sentinel: use "SYSTEM" (or a dedicated "ANONYMOUS" constant shared across iam) for the unauthenticated performedBy so audit consumers see one canonical value. No security impact.

### [NIT] RefreshToken.revoke timestamp source
- file: d:\My_Works\HMS\HMSCLEAN2\backend\src\main\java\com\otapp\hmis\iam\domain\RefreshToken.java 63-66
- issue: revoke() sets revokedAt = Instant.now() directly rather than using the injected Clock that AuthenticationService uses for all other time decisions (refresh/expiry). This means the forensic revokedAt timestamp is not aligned with the testable clock and is wall-clock-bound. The DB CHECK (revoked ↔ revoked_at) is satisfied either way, so this is not a correctness or security defect, just a minor forensic-timestamp/testability inconsistency on the reuse-detection audit fields (D-2).
- fix: Optional: pass the now Instant into revoke(Instant now) (or accept the clock-bound value) so the recorded revokedAt is consistent with the rotation timestamp under a fixed Clock. Low priority; no behavioural change to the security control.
