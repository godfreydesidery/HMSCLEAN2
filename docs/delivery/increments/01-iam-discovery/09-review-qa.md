# 09-review-qa

Verdict: REQUEST_CHANGES

The test suite is structurally sound — the right test files exist, ArchUnit gates are well-written, BCrypt strength is verified, privilege fixture is accurate (35 lines, correct split), UserNoFormatter golden master is thorough, and the auth three-way branch unit tests are complete with a fixed clock. mvn verify being green is not in dispute.

However, the suite has two BLOCKER gaps and six HIGH gaps that prevent release sign-off.

The two BLOCKERs are: (1) PersonnelExtensionLifecycleIT does not actually test any personnel extension creation or deactivation — the @BeforeEach is a stub with a comment saying it will seed roles but the body does nothing, so the three test methods only cover ADMIN which never triggers an extension; the syncExtensions logic for CLINICIAN, CASHIER, NURSE, PHARMACIST, STORE-PERSON, and MANAGER is entirely unexercised at the integration level. (2) RefreshReuseIT does not verify that reuse detection revokes all of the user's other live tokens — the security-critical cross-session revocation side-effect is untested at the IT level (the unit test covers it in isolation but the IT only checks the HTTP error type).

The six HIGH gaps are: cross-user 403 for token revoke (named in the spec but absent); sequential USR-000-002 golden-master IT (spec mandates exact value, not just regex); PUT /users and DELETE /users 403 tests missing; PUT /roles and DELETE /roles 403 tests missing plus the ROOT role delete guard untested; no audit_logs row assertion in any IT (explicit DoD item); blank-password-keeps-hash and non-blank-re-encodes IT cases absent.

Additionally: V5 has the cost-12 re-hash commented out as a TODO (D-8 is approved but unimplemented); the CORS disallowed-origin assertion passes if the server returns '*' (wildcard regression); the expired-token IT actually exercises the unknown-token branch (wrong branch); the PrivilegeSeedIT SQL regex may silently extract zero rows if V2 ULIDs contain lowercase characters.

The MANAGEMENT/MANAGER AMB-2 fix has no test locking the trigger name. Semgrep action version should be updated. Minor duplication in UserNoFormatterTest. TokenRevokeIT body assertion is too loose post-revoke.

## Findings

### [BLOCKER] Personnel extension lifecycle — CLINICIAN/CASHIER path unverified
- file: backend/src/test/java/com/otapp/hmis/iam/PersonnelExtensionLifecycleIT.java 55–76
- issue: The CLINICIAN and CASHIER reserved role names cannot be created via POST /iam/roles (the reserved-name guard blocks them), and the test acknowledges this problem in comments but then falls back to only testing ADMIN — which never triggers any extension. The comment says 'we seed them directly into the RoleRepository (test-scope, bypasses the reserved-name guard)' but the actual @BeforeEach method body sets rolesCreated=true and does nothing: no repository call is made. The three test methods test only: ADMIN creates no extension (trivially true), no-roles creates no extension (trivially true), and role-name lifecycle via ADMIN (not an extension trigger). The spec mandates: 'multi-role create → multiple extensions created and active; role removal → deactivation; one-per-type enforced'. None of those are asserted. The syncExtensions logic in UserAdminService is entirely untested at the integration level. This is a DoD gap for the most complex stateful logic in the increment.
- fix: Inject RoleRepository directly in the test (or add a @Sql script that inserts CLINICIAN and CASHIER rows directly, bypassing the controller guard), then write three tests: (1) create a user with roleNames=[CLINICIAN, CASHIER] → assert clinicianRepository.findByUser(user).get().isActive() is true AND cashierRepository.findByUser(user).get().isActive() is true; (2) update that user with roleNames=[CASHIER] → clinician extension is now deactivated (active=false), cashier remains active=true; (3) create two users with the same CLINICIAN role — assert that UNIQUE(user_id) on the clinicians table means each user gets exactly one extension row. The @BeforeEach can insert rows via jdbcTemplate or RoleRepository.save(new Role("CLINICIAN", "SYSTEM")) without going through the controller.

### [BLOCKER] Refresh reuse — all live tokens revoked assertion is missing in the IT
- file: backend/src/test/java/com/otapp/hmis/iam/RefreshReuseIT.java 38–59
- issue: The spec (build-spec §7, §4 branch 2) requires the reuse 401 to trigger revocation of ALL user's live tokens. The test checks that the error type is token-reuse-detected and the body has no hash/uid leakage — but never verifies that a second, independent refresh token for the same user is also invalidated after the reuse is detected. The revoke-all side-effect is the security-critical property of reuse detection (it is the mechanism that limits blast radius); testing only the error code does not prove it. The AuthenticationServiceTest unit test does cover this at the unit level for liveToken.isRevoked(), but the IT exercises the full HTTP stack and database and is the appropriate place to assert the cross-session revocation.
- fix: In RefreshReuseIT.reuseDetected_returns401WithTokenReuseDetectedType: after login, perform a second login to get a second independent refresh token for the same admin user. Rotate the first token once (consuming it). Present the first token again to trigger reuse. Then try to refresh using the SECOND (still-live) refresh token — assert it also returns 401 (it should be invalid-token or token-reuse-detected, either way unusable) because all live tokens were revoked by the reuse handler. This proves the revoke-all side-effect rather than just the HTTP status code.

### [HIGH] TokenRevokeIT — cross-user 403 is not tested
- file: backend/src/test/java/com/otapp/hmis/iam/TokenRevokeIT.java 1–120
- issue: The spec (build-spec §7, §2 endpoint #15) explicitly requires a test: 'revoke_crossUserToken_returns403'. TokenRevokeIT has revoke_returns204, revoke_withoutToken_returns401, revoke_unknownToken_returns204 (non-enumerating), and revoke_alreadyRevoked_returns204, but the cross-user ownership check test is absent. The production code (AuthenticationService.revoke) throws AccessDeniedException when the token belongs to a different user. This is a security property: a valid JWT holder must not be able to revoke another user's refresh token. Without the IT, the ownership gate could be silently removed and no test would fail.
- fix: Add revoke_crossUserToken_returns403: (1) login as admin to get admin's refresh token and access token; (2) create a second user, login as that second user to get their access token; (3) attempt POST /auth/token/revoke with the second user's access token but providing admin's refresh token in the body; (4) assert 403.

### [HIGH] UserAdminIT — sequential userNo golden master absent
- file: backend/src/test/java/com/otapp/hmis/iam/UserAdminIT.java 100–109
- issue: create_assignsUserNoInExpectedFormat only asserts the regex USR-\d{3}-\d{3}. The spec (build-spec §7, CR-06) mandates a 'sequential create-user IT proving USR-000-002 etc'. Because admin is already seeded with USR-000-001 and seq_usr_no is set to 1 via setval (so nextval returns 2), the first user created via the API should get USR-000-002. The test does not assert the exact value. This is the golden-master requirement for the numbering scheme — the regex passes even if the formatter produced USR-000-042 due to a seq_usr_no initialisation bug.
- fix: Add create_secondUserGetsUsrZeroZeroTwo: create one user (the first after admin, seq=2) and assert jsonPath('$.userNo').value('USR-000-002'). Because the Testcontainers PG is shared across the test class, this test is order-dependent. To make it deterministic: (a) use @TestMethodOrder(MethodOrderer.OrderAnnotation.class) and give this test @Order(1) before any other creation in the class, or (b) read the current max userNo from the list endpoint and assert the new user's userNo is exactly one seq step higher.

### [HIGH] UserAdminIT — update endpoint 403 / 401 tests absent
- file: backend/src/test/java/com/otapp/hmis/iam/UserAdminIT.java 1–217
- issue: The spec requires 'a negative authorization test (403) for EVERY mutating endpoint when the token lacks the required code; and 401 when unauthenticated on reads'. The create endpoint has both 401 and 403 tests. The PUT /users/uid/{uid} (update) and DELETE /users/uid/{uid} endpoints have no 403 test with an insufficient-privilege token. The AssignRolesIT covers assign-roles 403, but update and delete are missing it in UserAdminIT.
- fix: Add update_withoutRequiredPrivilege_returns403 and delete_withoutRequiredPrivilege_returns403 in UserAdminIT, mirroring the pattern of create_withoutRequiredPrivilege_returns403: mint a token with only DAY-ACCESS and attempt PUT/DELETE on any existing user uid, assert 403.

### [HIGH] RoleAdminIT — update and delete 403 / 401 tests absent
- file: backend/src/test/java/com/otapp/hmis/iam/RoleAdminIT.java 1–177
- issue: The spec requires negative auth tests for every mutating endpoint. RoleAdminIT has 403 for create and replacePrivileges, but PUT /roles/uid/{uid} (update) and DELETE /roles/uid/{uid} have no 403 (insufficient privilege) test. The delete endpoint also has no test for the ROOT guard (deleting the ROOT role should be rejected — the service enforces this but no IT exercises it).
- fix: Add: (1) update_withoutRequiredPrivilege_returns403 (token with DAY-ACCESS only → 403 on PUT /roles/uid/{uid}); (2) delete_withoutRequiredPrivilege_returns403 (token with DAY-ACCESS only → 403 on DELETE /roles/uid/{uid}); (3) delete_rootRole_isRejected (token with ROLE-ALL, attempt to delete the ADMIN role's uid which has owner=SYSTEM; or if ROOT exists, attempt to delete it → 4xx). The ROOT guard test is important parity coverage for the legacy UserResource.java:329.

### [HIGH] Audit trail — no IT asserts an audit_logs row is written per mutation

No test in any IT file queries the audit_logs table to assert that a record with the correct entityType ('iam.User'), entityUid (the created user's uid), action (CREATE), and actor (the JWT subject) was written. The AuditRecorder is called in every service method, but the integration-level assertion is absent. This is a DoD line item.
- fix: Inject JdbcTemplate (or an AuditLogRepository) into UserAdminIT. After create_withUserAll_returns201WithLocation, query: SELECT count(*) FROM audit_logs WHERE entity_type='iam.User' AND action='CREATE' and assert count >= 1. Optionally assert actor = 'admin' (the JWT subject) and entity_uid matches the uid in the Location header. Add equivalent assertions for at least one UPDATE and one DELETE operation.

### [HIGH] UserAdminIT — update blank-password-keeps-hash and non-blank re-encode tests absent
- file: backend/src/test/java/com/otapp/hmis/iam/UserAdminIT.java 1–217
- issue: The spec (build-spec §7) requires 'update_blankPasswordKeepsHash' and 'update_nonBlankPasswordReEncodes'. These are direct parity tests for the legacy UserServiceImpl lines 121–125 (blank → keep old password hash; non-blank → encode). Neither test exists. The production code in UserAdminService.update handles both branches (lines 163–169) but no IT verifies the blank-password case (which would manifest as the user being unable to log in if the hash is accidentally wiped) nor the non-blank re-encode case.
- fix: Add update_blankPassword_keepsHashAndLoginStillWorks: create a user with a known password, update them with password='', then verify login succeeds with the original password. Add update_nonBlankPassword_reEncodesAndNewPasswordWorks: create user with password 'OldPass1', update with password='NewPass1', assert login with 'OldPass1' returns 401 and login with 'NewPass1' returns 200.

### [HIGH] UserAdminIT — userNo immutability (code immutable) test absent
- file: backend/src/test/java/com/otapp/hmis/iam/UserAdminIT.java 1–217
- issue: The spec (build-spec §7) requires 'update_codeImmutable → 400/422 on userNo change attempt'. The production service does not expose userNo in the UpdateUserRequest DTO (it has no userNo field), so strictly the format prevents it from being changed. However the spec explicitly mandates this test exists to lock the invariant and to verify the immutability is enforced, not just omitted from the DTO by oversight. The legacy validation was explicit: 'Changing user code is not allowed' (UserServiceImpl line 119). No IT verifies the immutability at the HTTP level.
- fix: The simplest test is: create a user, capture their userNo, perform an update via PUT, then GET the user and assert the userNo in the response is identical to the one received at create time. This confirms the field is persisted immutably through the round-trip.

### [MEDIUM] PrivilegeSeedIT — fixture comparison is set-equality, not exact-match
- file: backend/src/test/java/com/otapp/hmis/iam/PrivilegeSeedIT.java 54–65
- issue: exactlyThirtyFiveCodesMatchFixture uses containsExactlyInAnyOrderElementsOf, which is set-equality (order-independent). This is correct. However, allSeededPrivilegeCodesArePresent (lines 38–51) derives its 'expected' set by parsing V2 SQL with a regex, then asserts the DB matches that parsed set. The regex pattern on line 101 — Pattern.compile("'[0-9A-Z]{26}',\\s*'([A-Z0-9_\\-]+)'\\s*,\\s*now\\(\\)") — uses [0-9A-Z]{26} to match the ULID prefix of each INSERT row. ULIDs in V2 contain lowercase characters, so this regex will match zero rows, making the test vacuously pass (the assertThat(expected).isNotEmpty() guard on line 45 would catch this, but only if the regex fails to match). This needs verification. If the V2 INSERT uses lowercase ULIDs (as ULID spec allows lowercase), the regex silently extracts nothing and the isNotEmpty assertion fails rather than giving a useful diagnostic.
- fix: Change the ULID portion of the regex to [0-9A-Za-z]{26} (case-insensitive ULID chars) and add a case-insensitive flag, or simply remove the ULID constraint entirely since you only need to extract the privilege code: use Pattern.compile("INSERT INTO privileges[\\s\\S]*?'([A-Z][A-Z0-9_-]+)'") with a broader context scan, or parse the SQL by splitting on commas and extracting the second quoted token per VALUES row.

### [MEDIUM] CORS test — disallowed-origin assertion is too weak
- file: backend/src/test/java/com/otapp/hmis/iam/CorsIT.java 36–46
- issue: disallowedOrigin_notEchoedInPreflightResponse asserts allowOrigin is not equal to 'https://evil.example.com'. This passes if allowOrigin is null (origin rejected, as expected) but it also passes if allowOrigin is '*' (wildcard — which would be a regression to the pre-D1 config). The spec (build-spec §4 D-1) mandates that the wildcard was removed. The test does not distinguish 'null / no header' from 'wildcard'.
- fix: Change the assertion to: assertThat(allowOrigin).isNull() for the disallowed case (Spring Security returns no Access-Control-Allow-Origin when the origin is not in the allowlist). Additionally add a test asserting that a wildcard is never returned for any origin: given 'http://localhost:4200' as origin, assert the returned header equals exactly 'http://localhost:4200' (already tested) AND is not '*'.

### [MEDIUM] RefreshReuseIT — expired-token branch uses a non-expired garbage token, not a genuinely expired one
- file: backend/src/test/java/com/otapp/hmis/iam/RefreshReuseIT.java 62–73
- issue: expiredToken_returns401InvalidToken_notReuseDetected presents 'this-token-never-existed-in-db' (an unknown token) which exercises Branch 1 (unknown hash), not Branch 3 (found + expired). The test name claims to verify the expired branch but actually verifies the unknown-token branch. The two are distinct: Branch 1 returns invalid-token because the hash lookup returns empty; Branch 3 returns invalid-token because the token exists but its expiresAt has passed. They share the same HTTP response but the code paths diverge. A future refactor that accidentally conflates the two branches would not be caught.
- fix: To test a genuinely expired token: (1) login to get a refresh token; (2) using JdbcTemplate, UPDATE refresh_tokens SET expires_at = now() - interval '1 hour' WHERE revoked = FALSE AND user_uid = (SELECT uid FROM users WHERE username='admin'); (3) present the raw refresh token to POST /auth/token/refresh; (4) assert 401 with urn:hmis:error:invalid-token and NOT urn:hmis:error:token-reuse-detected. This exercises Branch 3 and distinguishes it from Branch 1.

### [MEDIUM] UserDeleteIT — delete root-user guard test absent
- file: backend/src/test/java/com/otapp/hmis/iam/UserDeleteIT.java 1–97
- issue: The legacy-analyst confirmed the delete root guard (UserResource.java lines 197–201: reject if username.equalsIgnoreCase('root')). The production service (UserAdminService.delete line 227) reproduces this. UserDeleteIT does not test it. Given that the bootstrap principal is 'admin' (CR-20), there is no 'root' user in the DB to target. Alternatively, assert that deleting the seeded admin user (which plays the root role) is rejected — but the guard is on the string 'root', not 'admin', so this would only be meaningful if a root user is first inserted.

Then attempt DELETE /api/v1/iam/users/uid/{uid} and assert 422 (BUSINESS_RULE). This locks the invariant that the root guard works at the service layer, not just the controller layer.

### [MEDIUM] V5 migration — cost-12 re-hash is a TODO placeholder and not implemented

### [MEDIUM] Semgrep CI step references wrong path in workflow

- issue: The Semgrep action is configured with 'config: backend/ci/semgrep-secrets.yml'. - fix: Verify that returntocorp/semgrep-action@v1 is the current supported version (as of 2026 it is deprecated in favour of semgrep/semgrep-action). ### [NIT] TokenRevokeIT — body assertion after revoke is too loose
- file: backend/src/test/java/com/otapp/hmis/iam/TokenRevokeIT.java 63–65
