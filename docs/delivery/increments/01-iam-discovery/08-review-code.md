# 08-review-code

Verdict: REQUEST_CHANGES

Increment 01 (Identity & Access) is a strong, well-structured implementation that faithfully reproduces most of the legacy UserServiceImpl.saveUser / UserResource behaviour and correctly applies the ratified deviations (35-code seed with category tagging, USR-NNN-NNN via seq_usr_no, six personnel extensions, real delete + activate/deactivate, hardened privilege-assignment gate, plural tables, MapStruct mappers package-private with no passwordHash leak, no id in DTOs, no @Transactional on controllers, iam.lookup named interface with no entity leak, three-way refresh branch + reuse-detection, RFC7807 errors). The structural/security gates (PrivilegeGateArchTest, IamNoEntityLeakArchTest, ApiConventionsArchTest, Modulith verify, CORS lock-down, semgrep) are sound. However, there are exact-process fidelity defects that must be fixed before merge: (1) the reserved-role-name guard uses case-SENSITIVE Set.contains where legacy uses equalsIgnoreCase — a role named "clinician"/"Root" is rejected by legacy but allowed by the modern build, and the test suite only checks uppercase so it passes a false-green; (2) the role-update reserved guard checks the NEW requested name rather than the EXISTING role's name as legacy does, inverting the semantics (legacy blocks editing a reserved role; modern blocks renaming any role to a reserved name); (3) the headline new business rule — the six-extension create/activate/deactivate lifecycle — is effectively UNTESTED: PersonnelExtensionLifecycleIT has a no-op @BeforeEach and only asserts the negative path (ADMIN/no-roles create no extension), never the positive create-with-CLINICIAN+CASHIER, role-removal-deactivates, or one-per-type cases the build spec/ratified §D require. A green build therefore gives false confidence on the most complex extracted rule. Plus several MEDIUM items: new-user enabled default diverges from legacy (legacy creates inactive active=false; modern creates enabled=true) without explicit ratification; the "ALL" privilege-replace semantics are an unverified interpretation (Q4 still open — legacy per-object ALL granted only OBJECT-ALL, not all 35); and the ratified D-8 cost-12 admin re-hash is left commented-out in V5. None are PHI leaks or security regressions, and no billing/clinical numeric output is involved. Fix the case-insensitivity + update-guard drift, add real positive-path extension lifecycle coverage, and confirm/ratify the enabled-default and ALL-semantics questions with the legacy-analyst before merge.

## Findings

### [HIGH] Correctness / exact-process fidelity (reserved-role guard)
- file: D:\My_Works\HMS\HMSCLEAN2\backend\src\main\java\com\otapp\hmis\iam\application\RoleAdminService.java 55-59, 75, 114
- issue: Reserved-role-name comparison uses case-SENSITIVE Set.contains(request.name()). Legacy compares with equalsIgnoreCase (UserResource.java:217,239,245-262,274,296,302-316). Therefore legacy rejects role names like 'clinician', 'Root', 'Cashier' but the modern build ACCEPTS them, allowing creation of duplicate-by-case roles that collide with the reserved/personnel-trigger names. This is behaviour drift on a security-relevant guard. The class Javadoc compounds the error by asserting 'case-sensitive string equality (legacy behaviour)', which is factually wrong. RoleAdminIT only tests exact-uppercase names so the gap is a false-green.
- fix: Compare case-insensitively to match legacy: e.g. normalise request.name() (trim+uppercase) before checking against the reserved set, or use RESERVED_NAMES.stream().anyMatch(n -> n.equalsIgnoreCase(request.name())). Apply in both create() and update(). Add a parametrized test asserting lowercase/mixed-case reserved names are rejected. Correct the Javadoc.

### [HIGH] Correctness / exact-process fidelity (role update guard semantics)
- file: D:\My_Works\HMS\HMSCLEAN2\backend\src\main\java\com\otapp\hmis\iam\application\RoleAdminService.java 111-122
- issue: update() rejects when the NEW requested name is reserved. Legacy updateRole (UserResource.java:295-316) rejects when the EXISTING role being edited is reserved (it checks roleToUpdate.getName()). The semantics are inverted: legacy forbids editing a reserved/system role at all; the modern build permits renaming a reserved role to a non-reserved name (e.g. rename ADMIN -> ADMIN2) while forbidding renaming an ordinary role TO a reserved name. This both loses a legacy protection (mutating a system role) and adds a constraint legacy did not have.
- fix: Guard on the existing role's name as legacy does: if RESERVED (case-insensitive) contains role.getName(), reject the edit. Decide with engagement-lead whether to also keep a 'cannot rename TO reserved' check as an additional hardening (record as a deviation if kept). Add tests for: editing an existing reserved role is rejected.

### [HIGH] Test coverage / DoD (personnel-extension lifecycle)
- file: D:\My_Works\HMS\HMSCLEAN2\backend\src\test\java\com\otapp\hmis\iam\PersonnelExtensionLifecycleIT.java 54-76, 78-152
- issue: The core new business rule of this increment (UserAdminService.syncExtensions: create-or-reactivate on role assignment, symmetric deactivate on role removal, one-per-type — the faithful reproduction of legacy saveUser lines 138-357 with the ratified AMB-2/AMB-5 fixes) is effectively untested. The @BeforeEach is a no-op (just sets a flag, with a comment explaining it gives up on seeding the reserved roles), and all three test methods only assert the NEGATIVE path (ADMIN role / no roles => no extension). There is no test that a user created with CLINICIAN+CASHIER gets both extensions active, that removing a role deactivates that extension, that MANAGER triggers Management (AMB-2), or that all six deactivate symmetrically (AMB-5). The 'GREEN mvn verify' therefore does not exercise the highest-risk extracted logic.
- fix: Seed the reserved role names directly in the test (RoleRepository/JdbcTemplate, bypassing the API guard) or via a test-only V-migration/@Sql, then add real assertions: create with CLINICIAN+CASHIER -> clinicianRepository/cashierRepository.findByUser active=true; update removing CASHIER -> cashier inactive, clinician still active; create with MANAGER -> management active (AMB-2); verify one-per-type. This is required to discharge the build-spec §7 / ratified §D DoD item.

### [MEDIUM] Exact-process fidelity (new-user active/enabled default)
- file: D:\My_Works\HMS\HMSCLEAN2\backend\src\main\java\com\otapp\hmis\iam\domain\User.java 41-42
- issue: Modern User.enabled defaults to true, so a newly-created user can log in immediately. Legacy User.active defaults to false (legacy User.java:69) and saveUser's create branch (UserServiceImpl.java:107-115) never sets active=true (only the update branch line 126 does). Thus legacy-created users start INACTIVE and must be activated before login. This changes the create workflow. It is not covered by the ratified decisions (decision #3 ratifies real activate/deactivate but does not state the create-time default), and UserAdminIT's self-toggle/login tests implicitly assume the modern enabled=true default.
- fix: Escalate to legacy-analyst/engagement-lead: confirm whether new users should be created disabled (legacy-faithful) or enabled (modern). If kept as enabled=true, record it as a ratified deviation; if reverted to disabled, update create() and the dependent tests accordingly.

### [MEDIUM] Exact-process fidelity (privilege-replace 'ALL' semantics, Q4 open)
- file: D:\My_Works\HMS\HMSCLEAN2\backend\src\main\java\com\otapp\hmis\iam\application\RoleAdminService.java 136-152
- issue: replacePrivileges treats 'ALL' anywhere in privilegeCodes as 'grant every seeded privilege (35)'. Legacy addPrivilegeToRole (UserResource.java:444-485) used a per-object matrix where 'ALL' among an object's operations meant grant only the single OBJECT-ALL privilege code for that object, not all privileges. The build spec flagged this as open question Q4 and it is NOT resolved in 07-DECISIONS-RATIFIED. The request DTO shape was also changed from AccessModel{object,operations[]} to a flat privilegeCodes[], so the legacy semantics cannot be reproduced as-is — this is an unverified behavioural interpretation, not a confirmed match.
- fix: Escalate Q4 to legacy-analyst for the authoritative 'ALL' meaning. Document the chosen semantics in the spec/ADR and add a test pinning it. If 'ALL' -> all-35 is accepted, record it as a ratified deviation from the legacy per-object semantics.

### [MEDIUM] Security hardening (ratified D-8 not implemented)
- file: D:\My_Works\HMS\HMSCLEAN2\backend\src\main\resources\db\migration\V5__backfill_admin_identity.sql 8-17
- issue: Ratified decision D-8 (07-DECISIONS-RATIFIED §B) approves re-hashing the cost-10 admin seed to cost-12 in V5. The encoder is BCryptPasswordEncoder(12) (SecurityConfig.java:81) but V5 leaves the cost-12 re-hash entirely commented out as a TODO, so the admin row stays cost-10. Login still works (BCrypt embeds cost), so this is not a functional break, but a ratified strengthening is unshipped and the migration carries an unfilled placeholder.
- fix: Either implement the guarded cost-12 UPDATE in V5 (generate new BCryptPasswordEncoder(12).encode("password"), guarded by the exact old hash for idempotency) as D-8 directs, or explicitly defer D-8 with engagement-lead sign-off and remove the dangling commented block to avoid a half-applied migration.

### [MEDIUM] Exact-process fidelity (extension name/code refresh on reactivate)
- file: D:\My_Works\HMS\HMSCLEAN2\backend\src\main\java\com\otapp\hmis\iam\application\UserAdminService.java 258-370
- issue: On the reactivate branch (existing extension present), syncXxx only calls activate() and does not copyFrom(user), so a renamed user's existing extension keeps stale code/firstName/lastName/nickname. This matches the legacy reactivate behaviour (legacy only sets active=true on reactivate), but 07-DECISIONS-RATIFIED §C describes the lifecycle as 'create-or-reactivate ... (copy code + names, active=true)', which implies names should be refreshed on reactivate too. Ambiguous: faithful-to-legacy vs faithful-to-ratified-wording.
- fix: Clarify with BA/legacy-analyst whether reactivation should re-copy code+names. If yes, call copyFrom(user) (or a name-only refresh) before activate() in the reactivate branch; if no (legacy-faithful), tighten the ratified §C wording. Either way add a test pinning the chosen behaviour.

### [NIT] Defense-in-depth (update root guard)
- file: D:\My_Works\HMS\HMSCLEAN2\backend\src\main\java\com\otapp\hmis\iam\application\UserAdminService.java 148-195
- issue: update() has no root-specific guard, whereas legacy updateUser (UserResource.java:172-187) special-cases root (toggle-active-only, password forced blank, username-change rejected) and the ratified addendum says 'root/ROOT guards preserved'. This is moot in practice because create() rejects username 'root' and the bootstrap principal is admin/ADMIN (CR-20), so no 'root' row exists; delete() does preserve the root guard. Flagged only as defense-in-depth in case a 'root' row is ever introduced by a future migration/ETL.
- fix: Optionally add a symmetric 'cannot edit root profile' guard in update() mirroring delete(), so the invariant holds even if a root row is later introduced. Low priority.

### [NIT] Format fidelity edge case

### [NIT] Schema/entity consistency
- file: D:\My_Works\HMS\HMSCLEAN2\backend\src\main\java\com\otapp\hmis\iam\domain\User.java 49-50
- issue: userNo is mapped @Column(..., updatable = true) yet documented and enforced immutable only at the app layer (assignUserNo set-once + ck_users_user_no_format CHECK). There is no DB-level immutability; a stray UPDATE could change user_no. Functionally fine (the service is the only writer) but updatable=false would better encode the immutability invariant the build spec described.
- fix: Consider @Column(name="user_no", updatable=false). Note this would require V5 to set user_no before the entity ever updates the row (V5 already backfills via raw UPDATE, so app-layer updatable=false is safe). Low priority.
