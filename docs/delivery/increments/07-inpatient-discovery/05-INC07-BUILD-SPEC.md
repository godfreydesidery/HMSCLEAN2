## 2. Prerequisite — Cross-module seams (Spring Modulith named interfaces, ports/DTOs, allowed-edge list)

**Overview.** The cross-module wiring that MUST land before any inpatient domain/application code, mirroring the inc-08 `clinical::api`/`billing::api` reuse pattern, so `ApplicationModules.verify()` + ArchUnit stay green (no cycles).

**Allowed-dependency edge list for `inpatient`:** `{ "shared", "clinical :: api", "billing :: api", "masterdata :: lookup", "iam :: lookup" }`. NO reverse edge: `clinical`, `billing`, `registration`, `pharmacy`, `masterdata`, `iam` gain NO dependency on `inpatient`. The edge list deliberately OMITS `pharmacy`/`inventory` (Q12 billing-only — no stock decrement, no FEFO, no last-unit-409).

**Five seams, with decisions:**

1. **`AdmissionStatus` typed enum — OWNED by inpatient, no cross-module leak.** `inpatient.domain.AdmissionStatus` mapped 1:1 to legacy free-text strings via `@Converter` (ADR-0014 §1). Exactly five values: `PENDING("PENDING"), IN_PROCESS("IN-PROCESS"), STOPPED("STOPPED"), HELD("HELD"), SIGNED_OUT("SIGNED-OUT")` — hyphenated DB strings byte-preserved so 3 years of data migrate without value transformation. NO `TRANSFERRED` (Q8). Never crosses a module boundary.

2. **New published clinical seam `clinical::api.PrescriptionChartPort` — WRITE owned in `clinical`, called `inpatient -> clinical::api`.** The chart entity (V27) has REAL intra-module `@ManyToOne` FKs to `Prescription`/`Consultation`/`NonConsultation`, so the write is clinical-owned (mirrors inc-08 `pharmacy` orchestrating dispense while `clinical` owns the `Prescription` flip via `PrescriptionDispensePort`). Inpatient passes `admissionUid` + dosing fields into the port, which writes inside the caller's tx (`Propagation.REQUIRED`, no `@Async`/`REQUIRES_NEW`). **Guard split (confirmed, semantics + verbatim messages MUST be reproduced byte-identically across the boundary):** the admission-IN-PROCESS gate is evaluated in inpatient (it owns `AdmissionStatus`) BEFORE the call; the GIVEN-prescription guard, the exactly-one-encounter CHECK, and the nurse-uid guard are enforced clinical-side. The clinical port exposes uid-only DTOs (`PrescriptionChartView`, `RecordPrescriptionChartCommand`); NO `clinical.domain` entity crosses the boundary (ADR-0008).

3. **Bills-cleared discharge gate — NEW published admission-scoped billing query (IN-BASELINE-REQUIRED for parity).** The legacy gate scans `patientInvoiceRepository.findAllByAdmission(adm) -> PatientInvoiceDetail -> PatientBill.getStatus()` and rejects if ANY bill is `UNPAID` OR `VERIFIED` (`PatientResource.java:5342-5352` discharge / `:5593-5603` referral / `:5851-5882` deceased). `PatientInvoice` is billing-domain-owned; inpatient CANNOT enumerate the bills attached to the admission's invoices by OTHER modules (lab/radiology/procedure/pharmacy). Therefore the gate is reproduced by a NEW published billing seam (e.g. `BillingQueries.admissionHasOutstandingBills(admissionUid)` or `billStatusesForAdmission(admissionUid)`) that performs the `findAllByAdmission` scan INSIDE billing where `PatientInvoice` is owned, mapping legacy String statuses to `billing.domain.BillStatus` 1:1 (UNPAID/VERIFIED block; PAID/COVERED pass — byte-identical predicate). This is NOT optional and NOT a denormalised `billsCleared` flag. NOTE: the existing `BillingQueries.getBillStatus` is documented as scoped to the add-report bill-gate ONLY ("settlement gates MUST NOT use this") and is the WRONG seam for the discharge gate regardless of enumeration concerns.

4. **Payment-driven activation — NEW `inpatient.application.AdmissionSettlementListener` (resolution mechanism explicit).** `ConsultationSettlementListener` (clinical) silently ignores ward/registration bills, so inpatient adds its OWN `@TransactionalEventListener(phase = BEFORE_COMMIT)` on `shared.event.BillSettledEvent`. **Constraint:** `shared.event.BillSettledEvent` is `record(String billUid, Instant settledAt)` only — it carries NO `patientUid` and NO bill-kind, and `SettlementDispatcher` publishes it solely on CASH-bill PAID. The listener therefore resolves the admission by matching the ward-bed `billUid` (the uid persisted on the admission's `AdmissionBed.patientBillUid` at admit) OR via a billing seam resolving `billUid -> patientUid`. The legacy trigger (`PatientBillResource.java:352-365`) is PATIENT-SCOPED on the patient's bill payment (it promotes ALL of the patient's PENDING admissions; it does NOT filter on bill-kind='ward'); reproduce the trigger as legacy does. **Solution-architect escalation:** confirm whether the existing CASH-only `BillSettledEvent` publication even fires for the ward-bed activation case, or whether the event must be extended with `patientUid` (a seam change — NOT a free reuse). The listener imports only `shared.event` (no inpatient->billing reverse edge).

5. **Deceased-finalize publishes the EXISTING `PatientDeceasedEvent` from inpatient's own service (Q6).** The inpatient deceased-finalize service publishes `shared.event.PatientDeceasedEvent(patientUid, actorUsername)` via `ApplicationEventPublisher` in-tx; the existing `registration.PatientClosureListener` flips `Patient.type = DECEASED`. NO new event/listener/field, NO `inpatient -> registration` compile edge. Inpatient does NOT call clinical `ClosureService` (OPD-only).

6. **Consumable pricing seam (Q13) — OWNER-GATED + display-literals CR-blocked.** Reuses `BillingCommands.recordClinicalCharge(ChargeRequest, ctx)` with `kind = ServiceKind.MEDICINE`, `serviceUid = medicineUid`. The MEDICINE pricing PATH is freezable once the owner confirms Q13. The legacy literals require CR-07-Q13-billing-display (`ChargeRequest` has no `billItem`/`description`).
