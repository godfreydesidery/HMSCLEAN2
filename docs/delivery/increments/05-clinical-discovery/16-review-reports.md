## SECURITY — APPROVE_WITH_GAPS

Adversarial security review of inc-05 (clinical/OPD), the most PHI-dense context (SOAP notes, diagnoses, lab/radiology results, cause-of-death). The two specific concerns I was asked to verify are CLEAN: (1) the GlobalExceptionHandler PHI-echo on 500 is closed — the catch-all returns ErrorCode.INTERNAL.title() and logs ex server-side (GlobalExceptionHandler.java:107-112), and the no-route 404 is a clean ErrorCode.NOT_FOUND.title() via NoResourceFoundException/NoHandlerFoundException (GlobalExceptionHandler.java:78-81); (2) the audit trail is comprehensive — every state-mutating clinical action (death note save/approve, working/final diagnosis create+delete, order accept/reject/verify/collect/hold/delete, prescription dispense/delete, attachment add/delete, closure transitions) records an append-only SHA-256-checksummed AuditLog via AuditRecorder with the real authenticated actor (ctx.actorUsername()), satisfying ADR-0007 and CR-INC05-03. No PHI appears in any URL/query string (all params are ULIDs or status enums), no PHI is logged (only ULIDs at DEBUG), the radiology byte[] blob is excluded from every DTO (hasAttachment boolean only) with no download endpoint, and attachment DTOs are metadata-only. The V29-V37 patient_uid backfills are data-safe: each backfills from the live patient_id FK then enforces NOT NULL before dropping the column, so an unresolved patient reference aborts the migration rather than silently nulling a PHI link. The auth-only model (CR-INC05-02, not re-flagged) is correctly enforced — anyRequest().authenticated(), stateless, no clinical PHI in any permitAll path. Two genuine gaps remain, both rooted in the deferred-audit-scope decision (CR-INC05-03 explicitly deferred the covered-entity set to security-architect PHI classification): the cross-module Patient identity mutation to DECEASED is unaudited, and PHI-dense bulk/by-patient reads have no access log. Neither is a coding defect in what was built; both are scope items the audit-classification CR must close.

- RadiologyController.listAttachments (and LabTestController.listAttachments, LabTestController.java:311-316) returns the attachment list 'regardless' of status, with the javadoc explicitly stating the VERIFIED download gate (Radiology.canDownloadAttachment(), Radiology.java:627-629) is 'enforced by the client' (RadiologyController.java:292-294). Because the attachment DTOs are metadata-only (uid/name/fileName/createdAt — RadiologyAttachmentDto.java, LabTestAttachmentDto.java) and there is NO server download endpoint that streams bytes, this leaks only attachment filenames before a result is VERIFIED, not PHI content — so the exposure is bounded and LOW. The concern for the record: the canDownloadAttachment()/canDownload VERIFIED gate is a domain method that no server endpoint currently invokes; it lives only as client-enforced UI guidance. When a byte-streaming download endpoint is eventually added (currently DEFERRED per RadiologyDto.java:14), that endpoint MUST enforce status==VERIFIED server-side, or pre-verification radiology/lab images become retrievable by any authenticated user.
- **Fix:** Document as a binding security acceptance criterion for the deferred attachment-download endpoint: the server must gate the byte stream on parent status==VERIFIED (Radiology/LabTest.canDownloadAttachment()) — never rely on the client display gate. No change needed to the metadata-list endpoint in inc-05.
