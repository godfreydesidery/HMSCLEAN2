# 14-review-security

Verdict: APPROVE_WITH_NITS

Increment 02 (Master Data) backend PASSES security + module-boundary review. Every state-mutating masterdata endpoint (POST/PUT/DELETE across ~25 controllers) is gated with a method-level @PreAuthorize drawn exclusively from the 26 live privilege codes — ADMIN-ACCESS (dominant), SUPPLIER_PRICE_LIST-ALL (SupplierItemPrice write/update/delete), DAY-ACCESS (BusinessDay open/close). The CR-15 ungated->ADMIN-ACCESS tightenings (DEVIATION-1 Item, -2 ItemMedicineCoefficient, -3 Supplier, -4 ServicePrice/InsurancePlan) are correctly applied with no invented codes. @EnableMethodSecurity is active and .anyRequest().authenticated() makes reads authenticated-but-role-ungated (legacy-faithful); PrivilegeGateArchTest fails the build on any non-live code. The net-new CLINICIAN gate is a genuine, non-bypassable authorization check in ClinicClinicianService.affiliateClinician (asserts UserSummary.roleNames contains CLINICIAN before delegating, throws 403 CLINICIAN_ROLE_REQUIRED) layered on top of the ADMIN-ACCESS privilege gate. Module boundary is clean: masterdata depends only on iam::lookup named interface; iam stores opaque clinicUid strings via @ElementCollection (no FK, no reverse edge); IamNoEntityLeakArchTest enforces no iam.domain leak; PriceLookup/ServiceKind/ServicePriceResult exposed via masterdata::lookup. Audit coverage is complete — every create/update/delete/upsert (incl. affiliation CREATE/DELETE and ServicePrice CREATE) records via the append-only, Propagation.MANDATORY, SHA-256-checksummed AuditRecorder. No PHI in masterdata (reference data only); company-profile logo is BYTEA/byte[] (no large-object OID handle); UserSummary excludes password hash; no hardcoded secrets (JWT key from ${JWT_SECRET}); no Envers/@Audited/device-fingerprint reintroduced anywhere. Findings below are minor consistency/observation notes only — no gate gap, boundary violation, audit weakening, or PHI/secret exposure was found.

## Findings

### [NIT] RBAC / endpoint scope vs build-spec
- backend/src/main/java/com/otapp/hmis/masterdata/api/ServicePriceController.java 55-63
- issue: Build-spec §3 gate-map row reads "ServicePrice upsert/delete (the 14 ungated InsurancePlan coverage/price mutations)" and §4 references a delete path, but the controller exposes only POST (upsert) + GET list + GET /resolve — there is no DELETE endpoint for service_prices. ServicePriceService likewise has no delete method (only upsert/resolve/get/list). This is not a security defect (no ungated path exists), but the absence of the spec'd delete operation should be confirmed as an intentional scope deferral rather than an oversight.
- fix: Confirm with engagement-lead that ServicePrice delete is intentionally deferred (consistent with PriceLookup being storage-tier-only in inc-02). If so, add a one-line note to the gate map / decisions doc so the deferral is traceable; otherwise add a DELETE endpoint gated ADMIN-ACCESS with an AuditAction.DELETE record.

### [NIT] Audit semantics
- backend/src/main/java/com/otapp/hmis/masterdata/application/ClinicClinicianService.java 70, 93
- issue: Affiliation CREATE/DELETE audit events are recorded against entityType "masterdata.ClinicClinician" with uid = clinicUid only; the affiliated userUid (the actual subject of the affiliation change) is not captured in the audit row. Because the AuditLog schema records (entityType, entityUid, action, actor, occurredAt, checksum) with no before/after payload, an auditor reconstructing "which clinician was attached/detached from this clinic" cannot do so from the audit trail alone. Affiliation is an identity/access boundary change, so the subject identity is audit-relevant.
- fix: Acceptable for inc-02 given the current AuditLog shape, but flag to data-architect: the audit requirements doc should specify capturing the affiliation subject (userUid) for clinic-clinician mutations — e.g. via a composite entityUid (clinicUid:userUid) or a before/after detail column — when the audit schema gains a payload field. No code change required now.

### [NIT] Audit semantics
- backend/src/main/java/com/otapp/hmis/masterdata/application/InsurancePlanService.java 49-58
- issue: createWithProviderFromRequest is a public @Transactional service method recording AuditAction.CREATE, but it is not wired to any controller endpoint (InsurancePlanController exposes only the nested POST .../providers/{uid}/plans -> create(providerUid, request)). Dead/unreachable mutation method. Not a security exposure today (no ungated route reaches it), but an unreferenced public mutation method is a latent risk if a future endpoint binds to it without a gate.
- fix: Either remove createWithProviderFromRequest if it is genuinely unused, or document why it is retained (e.g. for a planned flat POST /insurance-plans endpoint). If retained, ensure any future controller binding carries the ADMIN-ACCESS gate.

### [NIT] Documentation consistency
- backend/src/main/java/com/otapp/hmis/iam/domain/Clinician.java 30, 84
- issue: Javadoc states the clinician_clinic_uids @ElementCollection table is created by "V10 migration" and the affiliation join table; the actual file is V10__iam_clinician_clinic_affiliation.sql, which matches — but the build-spec body (§4) had used V10 for a currency seed in an earlier draft numbering. The shipped V6..V14 numbering is internally consistent (V10 = clinician affiliation), so this is documentation drift between the spec draft and the as-built migrations, not a code defect. Worth a quick reconciliation note so reviewers don't chase a phantom mismatch.
- fix: Add a short "as-built migration map (V6..V14)" note to the increment decisions doc reconciling it against the build-spec §4 draft numbering. No migration change needed.

### [NIT] Audit minimisation (PHI principle)
- backend/src/main/java/com/otapp/hmis/masterdata/application/CompanyProfileService.java 89
- issue: CompanyProfileService.current() (the GET) records an AuditAction.READ audit event. This is extra coverage (a read audit on non-PHI reference data) and is harmless, but it is inconsistent with the rest of masterdata where reads are NOT audited. Confirm this READ-on-GET is intentional (e.g. anticipating future PHI access-log alignment) rather than copy-paste, so the audit-coverage scope stays deliberate and the eventual PHI access-log spec is applied uniformly, not ad hoc.
- fix: Confirm intent. If read-auditing is desired, capture the convention in the audit requirements doc (which reads are audited and why); if not, drop the READ record to keep masterdata read-audit behaviour uniform. Either way, no PHI is involved here so it is non-blocking.
