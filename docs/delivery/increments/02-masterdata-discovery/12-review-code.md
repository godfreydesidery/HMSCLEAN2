# 12-review-code

Verdict: REQUEST_CHANGES

Increment 02 (Master Data) is largely a faithful, well-disciplined reproduction of the legacy model. The highest-risk area — PriceLookup.resolve / ServicePrice — is correctly built as a STORAGE-TIER primitive only: plan-covered hit → cash fallback (plan NULL) → 422; service_uid NULL for REGISTRATION (CR-18) and =Clinic.uid for CONSULTATION; WardType-only WARD (CR-12); active/min/max/currency provably inert (verified by ServicePriceIT). The §2.3 ward top-up + per-service fallback asymmetry + consultation hard-fail are correctly DEFERRED to billing and NOT leaked into masterdata (legacy citations at PatientServiceImpl 597-601 and 1795+ verified honest). Coefficient math is correct: computeCoefficient(1,3)=0.333333 and the lossless convert(3,1,3)=1.000000; later increments are Javadoc-steered to convert(). Catalog faithfulness holds (cash price on Medicine.price/Item.selling_price_vat_incl, name-only LabTestTypeRange, no-ICD DiagnosisType, free-text category/uom, no ClinicType, WardBed.no not unique, no per-ward price). Affiliation (P4b) is clean: iam owns loose clinicUids (no FK, no iam→masterdata import), CLINICIAN gate returns 403, affiliate/remove idempotent. Conventions enforced by ArchUnit (no id in DTOs, no @Transactional on controllers, package-private MapStruct, no now() in masterdata internals, all @PreAuthorize codes ∈ the live set). However, three legacy/contract-fidelity issues block a clean approval, two of which touch insurance pricing and read access and per the engagement guardrails require explicit sign-off before merge. Recommend REQUEST_CHANGES: fix or ratify the ServicePrice admin lifecycle gap and the CompanyProfile read-gate deviation, and escalate the legacy price==0⇒covered=false rule to the legacy-analyst/healthcare-domain-expert.

## Findings

### [HIGH] Pricing / Insurance (ServicePrice admin lifecycle)
- backend/src/main/java/com/otapp/hmis/masterdata/api/ServicePriceController.java 55-95
- issue: ServicePrice exposes only POST (insert-or-409), GET list, and GET resolve. There is NO UPDATE (PUT) and NO DELETE endpoint, and ServicePriceService has no update/delete methods (grep returned no matches). Once a (plan_uid, kind, service_uid, currency) row exists, the second POST returns 409 and there is no API path to change its amount, toggle covered, or remove it. Build-spec §3 explicitly specifies 'ServicePrice upsert/delete (the 14 ungated InsurancePlan coverage/price mutations)'. Legacy InsurancePlanResource provides update_*_price_by_insurance (true in-place UPDATE, e.g. line 279 setPrice) and change_*_coverage (toggle, e.g. line 228 setCovered) for all 7 service kinds. The implementation therefore materially reduces admin capability to maintain insurance pricing versus both legacy and the binding contract. Because this touches insurance pricing maintenance, it falls under the guardrail requiring explicit sign-off.
- fix: Add an update path (true upsert: update the existing row's amount/covered/min/max when the composite key already exists, instead of 409) and a DELETE endpoint, both gated ADMIN-ACCESS, with golden-master tests; OR obtain an explicit ratified deviation (engagement-lead + healthcare-domain-expert) that insurance-price edit/removal is intentionally out of scope for inc-02 and deferred, and record it in 11-DECISIONS-RATIFIED.

### [HIGH] Pricing / Insurance (covered semantics on price write)
- backend/src/main/java/com/otapp/hmis/masterdata/application/ServicePriceService.java 49-73
- issue: Legacy couples price and coverage on the price-update path: in update_*_price_by_insurance the rule is price==0 => setCovered(false), price>0 keeps covered, price<0 rejected (InsurancePlanResource.java:274-279, mirrored across all 7 kinds). ServicePriceService.upsert persists the caller-supplied covered flag verbatim with no price/coverage coupling, so the 'zero price un-covers the plan for this service' behaviour is not reproduced anywhere in masterdata. Whether this is an admin-write rule that belongs in the ServicePrice write path or a legacy mechanic intentionally replaced by the new admin grid is ambiguous; the build-spec §2.3 deferred-to-billing list does NOT include it (it is a write-side, not resolve-time, rule). Per the no-invention guardrail this must not be resolved by assumption.
- fix: Escalate to legacy-analyst (source-of-truth) and healthcare-domain-expert to confirm whether price==0 => covered=false must be reproduced on the ServicePrice write path. If yes, implement and test it; if intentionally dropped/relocated, record the ratified deviation. Do not merge while the pricing-coverage semantics are undetermined.

### [HIGH] Identity & Access / Read gating (CompanyProfile)
- backend/src/main/java/com/otapp/hmis/masterdata/api/CompanyProfileController.java 40-44
- issue: GET /api/v1/masterdata/company-profile is gated @PreAuthorize("hasAnyAuthority('ADMIN-ACCESS')"), and CompanyProfileIT.get_withoutAdminAccess_returns403 enshrines it. This diverges from legacy CompanyProfileResource (GET /company_profile/get at line 53-55 has NO @PreAuthorize — JWT-only, role-ungated) and from build-spec §3 which states 'reads require a valid JWT but carry NO role gate (reproduce legacy: masterdata GETs are ungated-by-role)'. Every other masterdata GET (e.g. ClinicController list/getByUid) is correctly ungated, so this is an outlier. Company profile (logo, registration_fee, invoice/quotation notes, bank blocks) is broadly read by non-admin roles when rendering receipts/invoices in legacy; gating it ADMIN-ACCESS narrows access in a way that can break downstream cashier/clinical read flows. The controller Javadoc claims 'Gated ADMIN-ACCESS (legacy gate map §3)' but §3 actually mandates ungated reads, so the citation is incorrect. This is an unratified deviation that tightens access.
- fix: Remove the @PreAuthorize on CompanyProfileController.current() so the GET is JWT-only (matching legacy and build-spec §3), and update CompanyProfileIT accordingly; OR obtain an explicit ratified deviation (security-architect + engagement-lead) to gate the company-profile read, recorded in 11-DECISIONS-RATIFIED, and correct the Javadoc citation.

### [MEDIUM] Inventory / Procurement (active default drift)
- backend/src/main/resources/db/migration/V7__masterdata_inventory.sql 153-165, 181-190
- issue: items_suppliers.active and supplier_item_prices.active are DEFAULT FALSE, and the entity Java defaults are also false (ItemSupplier.java:61, SupplierItemPrice.java:37), with request records carrying primitive boolean active (defaults to false when omitted). Legacy ItemSupplier.active=true (ItemSupplier.java:49) and SupplierItemPrice.active=true (SupplierItemPrice.java:42). The migration comments explicitly acknowledge the drift ('legacy: true ... we follow the build-spec which says ... FALSE') but cite a non-binding 07-design-schema sketch; 11-DECISIONS-RATIFIED (binding) does not ratify this, and build-spec §1.2 lists 'active' for these tables without a FALSE default. This is a behaviour drift from legacy on a flag that governs which supplier prices/relationships are treated as active in later procurement/price-list flows.
- fix: Either revert the two defaults to TRUE to match legacy (and default request active to true), or obtain an explicit ratified deviation in 11-DECISIONS-RATIFIED. Note: catalogs start empty so no migrated rows are affected, which is why this is MEDIUM rather than HIGH.

### [MEDIUM] Inventory (coefficient conversion parity, forward-looking)
- backend/src/main/java/com/otapp/hmis/masterdata/application/ItemMedicineCoefficientService.java 137-154
- issue: convert(qty, medicineQty, itemQty) = qty*medicineQty/itemQty rounded HALF_UP scale 6 reproduces the cited parity case (3,1,3 -> 1.000000) and is the correct lossless approach. However, legacy applies storeSKUQty * coefficient where coefficient is a full-precision double medicineQty/itemQty (InternalOrderResource.java:595, StoreToPharmacyTOServiceImpl.java:424), so convert() is not bit-identical to legacy double arithmetic in all cases. More importantly, the 'use convert(), never the stored 6-dp coefficient column' rule is enforced only by Javadoc — there is no compile-time/ArchUnit guard preventing a later increment from multiplying by the rounded coefficient (which would yield 0.999999).
- fix: When the pharmacy/store transfer increments land, add a golden-master parity test and (ideally) an ArchUnit/lint guard that flags use of ItemMedicineCoefficient.getCoefficient() in quantity-conversion code paths, steering all conversions through convert(). No change required in inc-02 itself; track as residual technical debt for qa-test-engineer.

### [NIT] Pricing (ServicePrice REGISTRATION key integrity)
- backend/src/main/resources/db/migration/V9__masterdata_insurance.sql 113-114
- issue: ck_service_prices_service_uid CHECK (kind = 'REGISTRATION' OR service_uid IS NOT NULL) enforces non-REGISTRATION rows have a service_uid, but does NOT enforce that REGISTRATION rows have a NULL service_uid. ServicePriceService.upsert/ServicePriceRequest also do not reject a REGISTRATION row supplied with a non-null service_uid. PriceLookup.resolve always queries REGISTRATION with service_uid IS NULL (CR-18), so a mis-entered REGISTRATION row with a non-null service_uid would be silently unreachable rather than rejected.
- fix: Optionally tighten the CHECK to (kind <> 'REGISTRATION' OR service_uid IS NULL) AND (kind = 'REGISTRATION' OR service_uid IS NOT NULL), or validate it in the service, so a REGISTRATION row cannot be persisted with a service_uid. Low impact since the admin UI drives input and no existing behaviour is corrupted.
