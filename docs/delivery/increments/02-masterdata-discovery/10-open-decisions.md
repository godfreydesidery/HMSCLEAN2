# Consolidated open decisions (from extractions)

## Org units
- ClinicType: legacy has NO ClinicType enum or entity and Clinic has no type field (grep ClinicType|clinicType = 0 matches). If increment-02 needs a clinic type/category, this is net-new and requires an engagement-lead change request — do not invent silently.
- Clinic consultation fee is two-tier and per-process must be preserved exactly: cash = Clinic.consultationFee (Clinic.java:53-54); insurance = ConsultationInsurancePlan.consultationFee keyed by (clinic, insurancePlan, covered=true), with hard failure 'Plan not available for this clinic' when missing (PatientServiceImpl.java:597-601). A unified ServicePrice matrix (inc-02 proposal) must reproduce BOTH the per-clinic base fee AND the per-(clinic,plan) covered override, including the missing-plan exception.
- Ward pricing is per WardType ONLY (WardType.price; WardTypeInsurancePlan.price per (wardType, plan)). There is NO per-Ward and NO per-WardCategory price. Confirm with healthcare-domain-expert whether modernization intends to keep WardCategory as non-priced descriptive grouping only.
- No standalone Bed entity exists — only WardBed (physical bed master) and AdmissionBed (occupancy/charging transaction with linked PatientBill, openedAt/closedAt). Data-architect must preserve this two-entity split; do not collapse into a single Bed table without a change request.
- WardBed.no is NOT DB-unique (no unique constraint, WardBed.java:41-42). Decision: should the modern schema add a (ward_id, no) unique constraint? This would be a behavioural tightening — requires explicit approval; flag, do not silently improve.
- Clinic-clinician affiliation is owned EXCLUSIVELY by iam Clinician.clinics @ManyToMany (Clinician.java:69-71); Clinic.clinicians is the inverse (Clinic.java:57-60). There is NO masterdata-side clinic_clinician table and NO ClinicClinician entity in legacy (grep = 0 matches). The inc-02 'masterdata ClinicClinician table' is NET-NEW and would relocate join-table ownership from iam to masterdata — confirm with solution-architect/engagement-lead whether to keep ownership in iam (matching legacy) or introduce a masterdata-owned table (a structural change, not exact-process).
- Exact physical name of the legacy clinic↔clinician join table is NOT in source (owning side declares no @JoinTable; Hibernate default-derives it). Read it from the deployed legacy DDL / Flyway baseline before writing the migration — do not assume 'clinicians_clinics' vs 'clinician_clinics'. Same applies to the StorePerson↔Store join table (Store.storePersons inverse).
- Legacy money fields here are JPA double (Clinic.consultationFee, WardType.price, ConsultationInsurancePlan.consultationFee, WardTypeInsurancePlan.price). Per pre-approved scope these map to NUMERIC(19,2)/BigDecimal — confirmed allowed; no further decision needed beyond noting it in migration spec.
- Audit/device-binding: these masterdata entities introduce no @Audited annotation and no auth/device surface, consistent with the standing phantom-feature finding. The authoritative Envers/device-fingerprint confirmation must still be recorded in the IAM/security spec, not here.

## Inventory
- Medicine and Item are SEPARATE entities (no shared base, no discriminator). The modernization must decide whether to keep two tables (medicines + items) joined by item_medicine_coefficients, or unify — but per 'exact process' the two-table separation and the store-SKU↔pharmacy-SKU split is load-bearing behaviour and should be preserved unless engagement-lead approves a change.
- CASH price lives directly on the entity (Medicine.price; Item.costPriceVatIncl/sellingPriceVatIncl) — there is NO cash-price table. Confirm with data-architect that the target ServicePrice matrix (inc-02 spec) is intended ONLY for insurance-plan pricing and that the cash/default price stays as a column on the catalog entity, otherwise medicine cash dispensing (PatientServiceImpl:1534/2308/3422) breaks.
- Category and UOM are free-text Strings (Item.category/uom, Medicine.category default 'MEDICINE'/uom, Medicine.type comment '//ORAL, ETC') with NO entity or enum and NO validation. Decision for data-architect/business-analyst: introduce reference tables/enums in the target, or preserve free-text exactly? Introducing constraints is a behaviour change requiring engagement-lead approval.
- Coefficient conversion is INCONSISTENTLY applied in legacy: applied in InternalOrderResource:595 and StoreToPharmacyTOServiceImpl:424, but COMMENTED OUT in StoreToPharmacyRNServiceImpl:319-321, PharmacyToPharmacyRNServiceImpl:308-310, and InternalOrderResource:1245-1251 (raw qty used). Ambiguity for healthcare-domain-expert/business-analyst: is the receiving-note (RN) path SUPPOSED to skip conversion (i.e. RN qty is already in pharmacy units), or is this a latent bug? Do NOT silently resolve.
- Coefficient is derived as medicineQty/itemQty (ConversionCoefficientResource:95,101) — a division stored in a double. The target NUMERIC(19,6) for coefficient/qty is a modernization decision with NO legacy precision evidence (all legacy fields are plain double). Data-architect to confirm scale 6 is sufficient for the division result and that rounding rules match legacy double arithmetic for bit-comparable transfer quantities.
- ItemSupplier (cost prices) and SupplierItemPrice (price + terms) overlap functionally — two tables modelling supplier pricing for an item. Clarify with business-analyst which is authoritative for procurement/GRN before the data-architect designs the target supplier-price model.
- ItemSupplier has NO manual audit columns while every other inventory entity does — confirm this asymmetry is intentional and not a migration concern.
- @PreAuthorize gating: items/save has its annotation COMMENTED OUT (ItemResource:187, with a reference to PROCUREMENT-ACCESS which MEMORY lists as a DEAD code) and coefficient save is commented out (ConversionCoefficientResource:70, ROLE-CREATE — also dead). medicines/save uses live ADMIN-ACCESS (MedicineResource:103). Flag for security-architect: catalog/coefficient writes are effectively UNGATED in legacy; choosing a real gate code is a security decision, not a like-for-like reproduction.

## Pricing/Insurance
- minAmount/maxAmount in the proposed unified ServicePrice have NO legacy source — no min/max/copay/percentage/cap/deductible field exists on ANY pricing entity (grep of domain/ returned none). Co-pay is implemented ONLY as the ward principal+supplementary bill split. DECISION for engagement-lead/data-architect: keep these columns inert (nullable, never read) or drop them; populating them is invention and requires an explicit change request.
- currency column in the unified matrix has NO legacy source — all legacy amounts are plain double with no currency concept anywhere. DECISION: confirm a single fixed default currency for migration; currency must NOT drive any resolve behaviour.
- Ward pricing is NOT a single keyed lookup. It is a list-based eligibility algorithm (highest covered ward price, short-circuit on same-plan match) plus a principal/supplementary top-up co-pay bill when the eligible plan is a DIFFERENT (referral) plan and covered<cash (PatientServiceImpl.java:1799-1897). The unified matrix can store the rows but the resolve algorithm + PatientBill.principal/supplementary linkage must be reproduced in code. CONFIRM with healthcare-domain-expert the intended referral semantics (is selecting the highest-priced covered ward across all of a plan's covered ward types correct, or a legacy quirk?).
- LabTestPlanPrice (lab_test_plan_prices, nullable FKs, no covered field) is DEAD — no repository/service/resolve usage anywhere (grep returns only its own class). DECISION for data-migration-engineer: confirm whether any production rows exist and whether to migrate them at all, or drop the table entirely. Do NOT model it as live lab pricing — LabTestTypeInsurancePlan is the live table.
- Service-specific not-covered fallback asymmetry must be preserved exactly: consultation HARD-FAILS with 'Plan not available for this clinic' (PatientServiceImpl.java:599-601); lab/radiology/procedure/medicine fall back to cash with status VERIFIED ONLY for inpatients (a.isPresent(), e.g. :912-918); registration silently leaves the cash bill UNPAID (:321 block skipped). CONFIRM these divergent behaviours are intended business rules, not bugs.
- savePharmacySaleOrderDetail (PatientServiceImpl.java:3418-3442) charges CASH ONLY (Medicine.price * qty) with no insurance branch, unlike the prescription/dressing medicine paths which DO apply MedicineInsurancePlan. AMBIGUITY: is pharmacy OTC/sale-order intentionally cash-only, or is insurance coverage applied later in a separate step? Flag for business-analyst.
- Consultation insurance is keyed by Clinic (ConsultationInsurancePlan.clinic), and RegistrationInsurancePlan has NO service FK (plan-only). The unified matrix's serviceUid must therefore be NULLABLE (registration) and, for CONSULTATION, must reference the Clinic id — confirm with solution-architect/data-architect that kind+serviceUid covers this.
- Plans are resolved by NAME at point of care (PatientServiceImpl.java:530; PatientResource.java:297,360,384), and a covered=false placeholder row (price=0) is auto-created by setup endpoints (InsurancePlanResource.java:179-192). Migration must preserve name/code uniqueness and may carry many covered=false/price=0 placeholder rows that are functionally cash — confirm whether to migrate placeholders or synthesize them on demand.

## Stakeholders/System
- CURRENCY: No Currency entity, no currency field, no TZS/TSH literal exists anywhere in com.orbix.api (verified; the single ItemResource hit is a false-positive substring of setShortName). Money is bare 'double' with implicit single-currency. Decision for data-architect/engagement-lead: keep currency implicit (exact-process) OR introduce explicit Currency only via approved change request. Do NOT invent a Currency entity in masterdata silently.
- DOCUMENT-TYPE TABLE: No document_type table/entity exists. Prefixes are hardcoded String literals per service (only CompanyProfile.employeePrefix is data-driven). Decision: the Increment-02 proposal of document-type prefixes + DB SEQUENCEs is DRIFT — legacy uses application-level MAX(id)+1, not sequences. Confirm with engagement-lead whether to (a) faithfully reproduce MAX(id)+1 (exact process, but not concurrency-safe) or (b) treat sequence adoption as an approved data-model-only change (process-equivalent) under the double→BigDecimal-style pre-approval.
- SPTO/PPTO DRIFT + SPT COLLISION: Spec wants SPTO (store→pharmacy) and PPTO (pharmacy→pharmacy), but legacy emits the SAME literal 'SPT' for BOTH (StoreToPharmacyTOServiceImpl.java:438 and PharmacyToPharmacyTOServiceImpl.java:428). Because each uses its own table's MAX(id)+1, the two streams can produce identical document numbers distinguishable only by table. Decision for engagement-lead: reproduce 'SPT'-for-both (exact process, preserves the defect) OR approve a CR to split into SPTO/PPTO. Data-migration-engineer must be told existing data has SPT for both.
- RECEIVED-NOTE PREFIXES: StoreToPharmacy received note = 'PGRN' (StoreToPharmacyRNServiceImpl.java:339); PharmacyToPharmacy received note = 'PPRN' (PharmacyToPharmacyRNServiceImpl.java:328); PharmacyToStore RO = 'PSR'; PharmacyToPharmacy RO = 'PPR'. These are the actual legacy prefixes — verify the spec's intended prefixes against these.
- COUNTER CONCURRENCY/EMPTINESS: MAX(id)+1 is not collision-safe and not per-prefix monotonic (it is per-table on surrogate id). Empty table -> MAX returns null -> NPE swallowed by catch -> id stays 1L (first doc = '...-1'). Some docs (GRN, Employee, Patient) save a placeholder first then re-number, so the embedded number may not equal the final id. qa-test-engineer must cover these edge cases.
- PATIENT MRNO: PatientServiceImpl.java:250 builds 'MRNO/' + calendar-year + '/' + id; the year is cosmetic (id never resets) and the inline comment marks it as temporary ('this is only for starting'). Confirm with healthcare-domain-expert whether the year segment is a required business format or incidental.
- REGISTRATION FEE TYPE: CompanyProfile.registrationFee is double default 0 (CompanyProfile.java:77); migrate to NUMERIC(19,2) per pre-approved double->BigDecimal directive. Behaviour: regFee>0 -> registration PatientBill UNPAID; regFee==0 -> VERIFIED (PatientServiceImpl.java:273-277); INSURANCE patients override with RegistrationInsurancePlan.registrationFee.
- COMPANYPROFILE VALIDATION: validateCompany() always returns true (CompanyProfileServiceImpl.java:101-107) — no service-level validation; only JPA @NotBlank on companyName/contactName fires at persistence. Decision: whether modern build should keep zero service validation (exact process) or add validation requires a CR.
