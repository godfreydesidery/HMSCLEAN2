# Consolidated open decisions (from extractions)

## Invoice/Payment core
- INVOICE LIFECYCLE MISMATCH: Spec (inc-04) says PatientInvoice PENDING->PARTIALLY_PAID->PAID/CANCELLED. Legacy header only uses {PENDING, APPROVED} and is physically DELETED on full cancel; settlement tracked per-detail (detail.status='PAID') + running amountPaid sum, never on the header. The spec lifecycle is a modernization invention. RESOLUTION QUESTION (engagement-lead): is the new richer invoice lifecycle an approved change request, or must we reproduce the legacy PENDING/APPROVED + delete-on-cancel behaviour exactly?
- REFUND PATTERN MISMATCH: Spec says 'refund via signed/negative PatientPaymentDetail amount (no Refund entity)'. Legacy PatientPaymentDetail has NO amount field, so a negative-amount row is impossible. Real legacy refund = flip PatientPaymentDetail.status to 'REFUNDED' + create a positive-amount PatientCreditNote (status PENDING). Confirm with engagement-lead which mechanism the modern build must implement (legacy credit-note pattern vs. spec's signed-detail pattern).
- NO PARTIAL PAYMENT IN LEGACY: every bill is paid in full (paid=amount, balance=0) and tendered total must EXACTLY equal the sum of selected bills (double == ; InvalidOperationException on mismatch, PatientBillResource.java:389-391). There is no per-payment-line allocation and no change/overpayment handling. If the modern design wants partial payment or per-line allocation, that is NET-NEW and needs an approved change request.
- EXACT-EQUALITY FLOAT HAZARD: payment confirmation uses primitive double exact equality (amount != totalAmount). Under the directive to switch money to BigDecimal HALF_UP, define the comparison precisely (compareTo on scaled NUMERIC(19,2)). Confirm rounding semantics with data-architect; legacy has none.
- CASHIERSHIFT IS INVENTED: no CashierShift / OPEN-CLOSED / NO_OPEN_SHIFT 409 / EOD snapshot exists in legacy (Cashier is a person profile; reconciliation = collections table). Confirm whether the inc-04 CashierShift is an approved net-new feature or must be dropped to honour 'exact process'.
- INSURANCECLAIM LEDGER IS INVENTED: no InsuranceClaim entity and no SUBMITTED/SETTLED/REJECTED status machine exist in this legacy source tree. The only claim representation is PatientInvoice (PENDING/APPROVED) + PatientInvoiceDetail. Confirm the inc-04 InsuranceClaim ledger is net-new (not 'carried forward from prior-build V70').
- PAYMENT MODES: no PaymentType enum exists; paymentType is a free String only ever set to CASH or INSURANCE in code; DEBIT CARD/CREDIT CARD/MOBILE appear only in a source comment (PatientBill.java:57) and Collection.paymentChannel is hard-coded 'Cash'. Decide whether modern PaymentType enum should include only the two live values or the commented set.
- INVOICE-DELETE COUNT BUG: PatientResource.java:666-669 uses j=j++ which never increments, so the empty-check is always true and the parent PatientInvoice is ALWAYS deleted whenever any one detail is cancelled (even if other details remain). Decide whether to reproduce this bug or fix it (fix = change request; flag for qa-test-engineer + healthcare-domain-expert).
- DEAD INVOICE FIELDS: PatientInvoice.amountAllocated and amountUnallocated are declared but never written. Recommend dropping from the modern model (no behaviour to preserve); confirm with data-architect/data-migration-engineer that no report reads them (grep clean within com.orbix.api).
- PATIENTINVOICEDETAIL.patientBill mapping is internally inconsistent: @OneToOne optional=true vs @JoinColumn nullable=false (PatientInvoiceDetail.java:56-59). Decide the modern nullability; in practice a detail is always created with a bill, so nullable=false is the de-facto behaviour.
- PCN NUMBERING is app-level MAX(id)+1 with format PCN{yyyyMMdd}-{id}, no zero-padding, no fiscal reset, collision-prone under concurrency. DocumentNumberService (target) uses seq_pcn_no; confirm target format = PCN{yyyyMMdd}-{seq} matches and whether zero-padding is desired (legacy has none).
- PATIENTINVOICE.no = Math.random() placeholder then overwritten with id.toString() (two saves). The effective legacy invoice number is just the DB id as a string (no prefix/padding/fiscal-reset). Confirm modern invoice numbering scheme with data-architect; legacy has effectively none.
- PROCEDURE-CANCEL CREDIT-NOTE INCONSISTENCY: the credit-note block in one procedure-cancel path is commented out (PatientResource.java:3490-3494) while lab/radiology/consultation create credit notes. So a paid+canceled procedure may NOT generate a refund credit note. Flag to healthcare-domain-expert whether this asymmetry is intended.

## Cashier/Collection/EOD
- CONFIRMED NET-NEW: CashierShift OPEN->CLOSED does not exist in legacy. A full grep for shift/openShift/closeShift/cashUp/reconcil across com.orbix.api returns zero matches. The only temporal primitive is the single global Day entity (status STARTED->ENDED, DayRepository.java:17-18), which is system-wide, not per-cashier, and performs no snapshot. The spec's CashierShift and NO_OPEN_SHIFT 409 are spec inventions/net-new hardening. RESOLUTION QUESTION for engagement-lead: confirm CashierShift is an approved net-new feature (it changes process, not just design) rather than 'exact process' — if the legacy never gated payment on an open shift, introducing a hard NO_OPEN_SHIFT 409 is a behavioural change requiring explicit sign-off.
- CONFIRMED NET-NEW: There is NO payment gate on any open shift OR open day. dayService.getDay()/getCurrentBussinessDay() are used only to stamp createdOn, never as a guard (PatientBillResource.java:210,254,335). A payment can be recorded regardless of day status. The spec's pay gate has no legacy equivalent.
- Cashier IS the iam personnel/role extension (auto-provisioned on role 'CASHIER', UserServiceImpl.java:243-266), parallel to Nurse/Pharmacist. It is NOT used at payment time — collections/payments are attributed to the logged-in User (created_by_user_id), and the per-cashier report keys on users.nickname (CollectionRepository.java:54), NOT the cashiers table. DECISION for data-architect/iam: model Cashier under iam personnel, and base any 'cashier' reporting on the user, not a Cashier FK, to match legacy.
- Collection is a per-bill-line write-once cash-receipt ledger row (no status, optional PatientBill FK), NOT a shift or persisted daily aggregate. EOD cash-up is computed at READ time via SUM(amount) GROUP BY itemName, paymentChannel (CollectionRepository.java:21-59). DECISION: if increment-04 wants a persisted EOD snapshot, that is net-new; the exact-process equivalent is the on-demand SUM report.
- AMBIGUITY / latent dead capability: paymentChannel is hardcoded 'Cash' at all three creation sites (PatientBillResource.java:206,250,331) and paymentReferenceNo always 'NA', yet the report GROUP BY paymentChannel implies multi-mode support. So legacy collections are single-mode in practice. RESOLUTION QUESTION for business-analyst/healthcare-domain-expert: is multi-mode payment (cash/mobile/card/insurance) a required NEW capability, or must the rebuild faithfully reproduce cash-only collections? This materially affects the PatientPayment/Collection model.
- AMBIGUITY: legacy money is double (Collection.amount, PatientBill.amount). Per project directive money becomes NUMERIC(19,2) BigDecimal HALF_UP (pre-approved). Flag for data-migration-engineer: the SUM aggregation in the cash-up report currently sums doubles; rebuilt report must SUM BigDecimal to remain bit-identical.
- BUG to flag for qa-test-engineer (do not silently fix — exact-process): the per-kind collection detail reports compare boxed Long bill ids with == identity (e.g. ReportResource.java:705,751,797,843,889,933,978) rather than .equals(); behaviour depends on Long cache (-128..127) and may mis-match for large ids. Reproduce-or-fix is a change-request decision for engagement-lead.
- The 'general' collections report (getCollectionReportGeneral) GROUPs BY (itemName, paymentChannel) only and does NOT group by user despite selecting users.id (CollectionRepository.java:21-39) — so it aggregates across all cashiers/users. The per-cashier variant filters by users.nickname. Both use date range from.atStartOfDay()..to.atStartOfDay().plusDays(1). Document this exact range semantics for the rebuilt report.
- Out-of-scope confirmations for this slice (defer to dedicated specs): full-codebase @Audited scan (audit-trail status) and device-fingerprint scan — within the cashier/collection entities examined, neither is present, consistent with the standing 'phantom features' finding (no Envers active, no device-binding).

## CreditNote/Numbering/Refund
- INVENTED SPEC vs LEGACY — credit-note write-off: Inc-04 assumes partial/full write-off that reduces invoice balance and sets invoice CANCELLED on full write-off. Legacy has NONE of this: PCN is always full bill amount, has no FK to invoice/bill, and never touches invoice balance or status. Confirm with engagement-lead whether to (a) reproduce legacy exactly (PCN = standalone PENDING record only) or (b) treat balance-aware write-off as an approved change request.
- INVENTED SPEC vs LEGACY — invoice lifecycle: Inc-04 PatientInvoice PENDING->PARTIALLY_PAID->PAID/CANCELLED is not in legacy. Legacy invoice header stays default PENDING (PatientInvoice.java:50); only details flip to PAID and amountPaid is incremented (PatientBillResource.java:345-348); amountAllocated/amountUnallocated are declared-but-never-written dead fields. Decide whether to introduce the lifecycle as new hardening (needs change request).
- INVENTED SPEC vs LEGACY — refund via negative PatientPaymentDetail amount: impossible in legacy because PatientPaymentDetail has no amount column (PatientPaymentDetail.java:37-58). Legacy refund = status flag REFUNDED (consultation, PatientResource.java:638) or hard-delete of payment detail + bill (lab/radiology/procedure/prescription/sale). Confirm whether the modern design may add a signed-amount refund line (change request) or must mirror the flag/delete approach.
- totalPaid / amountPaid are NEVER decremented on refund or cancel (PatientPayment.amount set once PatientBillResource.java:164/278; PatientInvoice.amountPaid only incremented :347). This can leave totals overstated after a refund. Ask business-analyst: is the overstated-paid behaviour intentional legacy process to reproduce, or a bug to fix under change request?
- LATENT BUG to confirm reproduction policy — j = j++ no-op (PatientResource.java:666-669, 2947-2950, 3453-3456): the empty-invoice check is always 0, so the PARENT PatientInvoice is deleted whenever ANY single detail is removed, even if other details remain. Intended behaviour was 'delete invoice only when zero details remain'. Decide: reproduce the buggy always-delete, or implement the intended guard (change request).
- DIVERGENT cancel behaviours: consultation cancel RETAINS bill (status CANCELED) and payment detail (status REFUNDED); lab/radiology/procedure/prescription/sale cancel HARD-DELETE bill + payment detail. Both emit a PCN. Confirm with healthcare-domain-expert which behaviour is canonical, or whether both must be preserved per source object.
- PCN date timezone: legacy number date = UTC (Formater uses LocalDateTime.now(); MainApplication.java:138 forces UTC), but createdAt = UTC+3 EAT (DayServiceImpl.java:87). Spec target wants PCN{yyyyMMdd} in EAT. Confirm: should the rebuilt DocumentNumberService use EAT for the number's date (fixing the UTC/EAT split), or faithfully reproduce the UTC date string? This changes the calendar date in the number for events between 00:00-03:00 EAT.
- PCN numbering concurrency: MAX(id)+1 computed before insert + unique constraint on no => concurrent cancellations can collide and the second insert throws a unique violation. Recommend DocumentNumberService use a DB sequence (seq_pcn_no already created in V13) keyed to insert, but VERIFY this is acceptable as a non-behavioural change (number FORMAT preserved: PCN{yyyyMMdd}-{seq}).
- PREFIX collision: 'SPT' is emitted by BOTH StoreToPharmacyTOServiceImpl.java:438 and PharmacyToPharmacyTOServiceImpl.java:428 for two different document types. If DocumentNumberService keys counters by prefix, this would merge two streams. Confirm whether each document type must keep an independent counter (recommended) even though legacy keys by entity table (MAX of each table independently, so legacy does NOT actually collide on id — only the human-readable prefix string overlaps).
- No PCN HTTP endpoint and no PCN read/list/approve flow exist in legacy (PatientCreditNoteService exposes only requestPatientCreditNoteNo). Confirm whether the modern billing module needs PCN query/approval endpoints (net-new) or PCN remains a write-only side-effect of cancellations.
- PatientCreditNote.patient is nullable (PatientCreditNote.java:46-49) though all current creation sites set it. Decide if the rebuilt schema enforces NOT NULL (tightening) or preserves nullable for migration fidelity.
